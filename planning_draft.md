좋습니다. 이건 그냥 “대시보드 몇 개 만들자” 수준이 아니라,
**정부·공공기관용 챗봇 + RAG 운영 + 고객사 어드민**을 하나의 운영 체계로 설계해야 합니다.

핵심은 두 축입니다.

* **운영사 관점**: 시스템을 안정적이고 효율적으로 운영해야 함
* **고객사 관점**: 챗봇이 실제로 민원응대 효율을 얼마나 높였는지 보여줘야 함

그래서 설계도 **기술 운영 지표**와 **업무 성과 지표**를 분리하되, 마지막엔 하나로 연결돼야 합니다.

아래처럼 단계별로 정리하겠습니다.

---

# 1. 먼저 목표를 분리해서 정의해야 함

## 1-1. 운영사의 목표

운영사는 보통 이런 걸 봅니다.

* 장애 없이 안정적으로 서비스 제공
* RAG 품질 유지
* 고객사별 운영 효율화
* 운영 인력 대비 많은 기관을 관리 가능하게 만들기
* 문제 발생 시 빠른 원인 파악 및 조치

즉 운영사는 **“시스템이 잘 돌고 있는가”**를 봐야 합니다.

## 1-2. 고객사의 목표

고객사는 이런 걸 봅니다.

* 민원 문의를 얼마나 자동 응대한 건지
* 상담원/콜센터 업무가 얼마나 줄었는지
* 답변 정확도와 만족도가 어떤지
* 어떤 민원이 많이 들어오는지
* 어떤 문서는 부족해서 챗봇이 답을 못하는지

즉 고객사는 **“업무 성과가 났는가”**를 봐야 합니다.

---

# 2. 전체 구조를 3개 어드민으로 나누는 게 좋음

권장 구조는 아래와 같습니다.

## A. 운영사 통합 어드민

운영사가 전체 고객사/기관을 한 번에 관리하는 화면

역할:

* 전체 서비스 상태 모니터링
* 기관별 장애/품질 이슈 탐지
* RAG 인덱싱 상태 관리
* 운영 이력/배포/정책 관리
* SLA 관리

## B. 고객사 기관 어드민

각 공공기관 담당자가 자기 기관 챗봇 성과와 운영 현황을 보는 화면

역할:

* 민원응대 효율 확인
* 자주 묻는 질문과 실패 질문 확인
* 문서 최신화 필요 영역 확인
* 상담 전환이 많은 주제 파악
* 부서별/업무별 성과 확인

## C. 품질관리/지식관리 어드민

운영사와 고객사가 함께 쓰거나 권한 분리해서 쓰는 화면

역할:

* 잘못된 답변 검수
* 미응답/저신뢰 질문 검토
* 문서 추가/수정 요청
* FAQ/정책문서/고시/민원업무 문서 연계
* RAG 데이터셋 품질 개선

이 3개를 분리하면 좋습니다.
특히 공공기관은 권한과 책임이 명확해야 하므로, **운영 화면과 성과 화면을 섞지 않는 것**이 중요합니다.

---

# 3. 어떤 지표를 쌓아야 하는가: 6개 레이어로 나눠야 함

지표는 무조건 많이 쌓는다고 좋은 게 아닙니다.
**의사결정에 필요한 층위로 나눠서 적재**해야 합니다.

---

## 3-1. 서비스 이용 지표

이건 “얼마나 쓰고 있는가”를 보는 가장 기본 지표입니다.

쌓아야 할 지표:

* 총 대화 수
* 총 질문 수
* 순 사용자 수(UV), 재방문 사용자 수
* 일/주/월별 이용 추이
* 시간대별 이용량
* 채널별 이용량(웹, 모바일, 카카오, 앱 내 챗봇 등)
* 기관/부서/업무 카테고리별 이용량
* 신규 질문 비율 vs 반복 질문 비율

왜 중요하냐면:

* 고객사는 “도입했는데 실제 사용되나?”를 궁금해하고
* 운영사는 “트래픽 급증/급감이 이상 징후인지”를 봐야 하기 때문입니다.

---

## 3-2. 민원응대 성과 지표

이게 고객사에게 가장 중요한 핵심 지표입니다.

쌓아야 할 지표:

* **자동응대 완료율**: 사람 연결 없이 챗봇으로 종료된 비율
* **상담 전환율**: 상담원/콜센터/민원창구로 넘어간 비율
* **1회 해결률(First Contact Resolution)**: 한 번의 대화에서 해결된 비율
* **평균 대화 턴 수**
* **평균 해결 시간**
* **재문의율**: 동일/유사 질문이 다시 들어오는 비율
* **포기율**: 답변 도중 이탈한 비율
* **민원 유형별 해결률**
* **업무별 절감 효과**: 전화 감소, 단순문의 감소, 상담 처리 시간 감소 추정
* **업무시간 외 응대 비율**: 야간/주말에 챗봇이 처리한 문의 비율

왜 중요하냐면:
고객사가 결국 보고 싶은 건 기술이 아니라
**“민원 대응이 빨라졌는지, 상담 부담이 줄었는지”**이기 때문입니다.

---

## 3-3. 답변 품질 지표

RAG 챗봇은 여기서 승부가 납니다.

쌓아야 할 지표:

* 답변 성공률
* 미응답률
* 불충분 응답률
* 잘못된 답변 신고율
* 사용자 만족도(좋아요/싫어요, 5점 척도 등)
* 근거 제시율(출처 문서 포함 비율)
* 출처 클릭률
* 답변 신뢰도 점수 평균
* 질문 의도 분류 성공률
* 정책/행정 용어 처리 정확도
* 금칙어/부적절 응답 탐지 건수
* 환각 의심 응답 비율

추가로 공공기관에선 꼭 봐야 하는 것:

* **최신 문서 기준 답변 비율**
* **법령/지침 변경 후 구버전 참조 비율**
* **기관 기준에 맞지 않는 표현 비율**
* **민감 민원(복지, 세금, 인허가 등) 오류율**

---

## 3-4. RAG 운영 지표

이건 운영사가 반드시 봐야 합니다.

쌓아야 할 지표:

* 문서 수집 건수
* 인덱싱 성공률 / 실패율
* 최신화 주기 준수율
* 문서 버전 수
* 문서별 참조 빈도
* 검색 Top-k 문서 노출 빈도
* 검색 성공률
* 검색 무결과율
* 검색 결과 재랭킹 점수 분포
* 질문-문서 매칭 정확도
* 오래된 문서 참조 비율
* 중복 문서 비율
* chunk 크기별 성능 비교
* 임베딩 생성 실패율
* 문서 파싱 실패율(PDF, 한글 문서, 표 문서 등)
* 메타데이터 누락률
* 기관별 지식베이스 커버리지

특히 중요:

* **질문은 많은데 답변 실패가 높은 카테고리**
* **문서가 있는데 못 찾는 경우**
* **문서 자체가 없는 경우**
* **검색은 됐는데 생성 답변이 이상한 경우**

이 세 가지를 분리해 보여줘야 개선이 가능합니다.

---

## 3-5. 시스템 운영 지표

이건 인프라/SRE/운영사 관점입니다.

쌓아야 할 지표:

* API 응답 시간
* LLM 응답 시간
* 검색 응답 시간
* 문서 인덱싱 처리 시간
* 요청 성공률 / 실패율
* 에러율(4xx, 5xx, timeout)
* 세션 생성 실패율
* 인증 실패율
* 채널 연동 장애율
* 서버/DB/벡터DB 상태
* CPU, 메모리, 저장공간
* 배치 성공률
* 외부 연동 상태
* 모델별 사용량
* 토큰 사용량
* 기관별 비용
* SLA 준수율
* 장애 발생 건수 / 평균 복구 시간(MTTR)

이건 고객사에 전부 보여줄 필요는 없고, 운영사 어드민 중심으로 제공하면 됩니다.

---

## 3-6. 운영 업무 지표

운영 효율을 위해 꼭 필요하지만 자주 빠지는 부분입니다.

쌓아야 할 지표:

* 기관별 운영 티켓 수
* 이슈 유형별 건수(오답, 미응답, 인덱싱, 권한, 채널, UI)
* 처리 소요 시간
* 재발 이슈 비율
* 담당자별 처리량
* 고객사 요청 반영 리드타임
* 검수 대기 건수
* 미반영 문서 수
* 개선 요청 후 품질 향상률
* 월별 운영 공수

이게 있어야 운영사가 “사람을 더 넣어야 하나, 자동화를 더 해야 하나”를 판단할 수 있습니다.

---

# 4. 어떤 뷰를 제공해야 하는가

이제 핵심입니다.
지표를 쌓는 것보다 더 중요한 건 **누구에게 어떤 화면으로 보여줄지**입니다.

---

## 4-1. 운영사 통합 대시보드

운영사가 첫 화면에서 봐야 하는 것

### 필수 카드

* 전체 기관 수 / 운영 중 기관 수
* 오늘 총 질의 수
* 평균 응답 시간
* 오늘 장애 건수
* 인덱싱 실패 건수
* 미응답 급증 기관 수
* 만족도 하락 기관 수

### 필수 뷰

1. **기관별 헬스맵**

   * 기관별 상태를 색상으로 표시
   * 정상 / 주의 / 위험
   * 기준: 에러율, 미응답률, 응답시간, 만족도 급락

2. **이상탐지 뷰**

   * 전주 대비 질문 급증
   * 특정 기관 미응답 급증
   * 특정 문서군 참조 급감
   * 상담 전환율 급등

3. **인덱싱 운영 뷰**

   * 수집 → 파싱 → 임베딩 → 인덱싱 단계별 성공/실패
   * 실패 문서 원인 분류
   * 최신화 누락 문서 목록

4. **장애/이슈 워룸 뷰**

   * 현재 오픈 이슈
   * 영향 기관 수
   * 장애 지속 시간
   * 최근 배포와 상관관계

5. **비용/사용량 뷰**

   * 기관별 토큰 사용량
   * 모델별 사용량
   * 질문당 평균 비용
   * 피크 시간대 비용

운영사는 이걸 보고 바로 조치할 수 있어야 하므로
**예쁜 리포트보다 액션 가능한 화면**이 중요합니다.

---

## 4-2. 고객사 기관 대시보드

고객사가 가장 먼저 봐야 하는 화면

### 첫 화면 핵심 카드

* 이번 달 총 문의 수
* 자동응대 완료율
* 상담 전환율
* 평균 응답 시간
* 사용자 만족도
* 절감 추정 문의 수

### 필수 뷰

1. **민원응대 성과 요약 뷰**

   * 전월 대비 문의량
   * 자동 해결 비율
   * 업무시간 외 처리량
   * 많이 처리된 민원 카테고리

2. **민원 유형 분석 뷰**

   * 상위 질문 주제
   * 카테고리별 해결률
   * 상담 전환이 많은 주제
   * 계절성/이슈성 민원 변화

3. **실패/미흡 응답 뷰**

   * 답변 실패 질문 목록
   * 만족도 낮은 질문
   * 출처 부족 질문
   * 정책상 민감 질문

4. **지식 공백 뷰**

   * 문서 없음
   * 문서는 있으나 검색 실패
   * 검색은 됐으나 답변 품질 낮음
   * 추가 FAQ 필요 항목

5. **채널별 성과 뷰**

   * 웹/모바일/콜봇 등 채널별 성과
   * 채널별 해결률, 이탈률, 만족도

6. **기간 비교 뷰**

   * 전주/전월/분기 비교
   * 개선 전후 비교
   * 문서 추가 이후 성과 변화

고객사는 기술보다 **성과와 개선 포인트**를 원합니다.
그래서 운영 지표를 많이 보여주기보다
“어떤 민원이 잘 해결되고, 어디가 약한지”가 더 중요합니다.

---

## 4-3. 품질 검수 뷰

이건 실제 운영에서 매우 중요합니다.

### 꼭 있어야 하는 화면

* 저평가 답변 리스트
* 미응답 질문 리스트
* 환각 의심 답변 리스트
* 상담 전환 다발 질문
* 출처 없는 답변 리스트
* 최신 문서 미반영 의심 질문
* 기관 담당자 검수 큐

### 액션 기능

* 정답/오답 라벨링
* 문서 추가 요청
* FAQ 등록
* 금칙 응답 패턴 등록
* 재학습/재인덱싱 요청
* “이 답변은 사용 금지” 처리
* 유사 질문 묶기

이 화면이 좋아야 품질 고도화 속도가 빨라집니다.

---

# 5. 공공기관용이라면 꼭 추가해야 할 지표/뷰

정부·공공기관은 일반 기업 챗봇과 다릅니다.
그래서 아래 항목은 반드시 고려해야 합니다.

## 5-1. 최신성 관리

법령, 조례, 지침, 공고, 민원처리 기준은 바뀔 수 있습니다.

필수 지표:

* 문서 최신 업데이트 일자
* 최신 버전 반영 여부
* 폐기 문서 참조 비율
* 변경 후 반영 소요 시간

필수 뷰:

* 문서 버전 이력
* 최근 개정 문서와 영향 질문
* 최신성 위험 알림

## 5-2. 설명 가능성

공공기관 답변은 근거 제시가 중요합니다.

필수 지표:

* 출처 포함 답변 비율
* 출처 클릭률
* 출처 없는 답변 비율
* 답변 근거 신뢰등급

필수 뷰:

* 답변별 참조 문서
* 문서 문단 매핑
* 답변 근거 미리보기

## 5-3. 민감업무 통제

복지, 세금, 민원 신고, 자격요건 등은 오류 비용이 큽니다.

필수 지표:

* 고위험 민원 카테고리 응답 건수
* 고위험 카테고리 상담 전환율
* 정책상 제한 응답 발생 건수
* 확인 필요 안내 포함 비율

필수 뷰:

* 고위험 질문 전용 검수 대시보드
* 민감 카테고리 별도 승인 워크플로우
* 책임 문구/유의문구 적용 현황

# 6. 실제로는 “원인 분류” 체계를 먼저 만들어야 함

많은 팀이 지표를 쌓아도 개선이 안 되는 이유는
문제를 분류하지 않기 때문입니다.

실패 질문은 아래처럼 원인 코드가 있어야 합니다.

* A01: 문서 없음
* A02: 문서는 있으나 최신 아님
* A03: 파싱 실패
* A04: 검색 실패
* A05: 재랭킹 실패
* A06: 생성 답변 왜곡
* A07: 질문 의도 분류 실패
* A08: 정책상 답변 제한
* A09: 사용자 표현이 너무 모호함
* A10: 채널 UI/입력 문제

이 원인 코드가 있어야

* 운영사는 시스템 문제를 고치고
* 고객사는 문서를 보강하고
* 둘 다 역할을 나눠서 개선할 수 있습니다.

# 7. 단계별 구축 로드맵

이제 실제 기획 순서로 정리해보겠습니다.

## 1단계: 최소 운영 체계 구축

목표: 서비스가 돌아가고, 기본 성과가 보이게

먼저 쌓아야 할 것:

* 질문/답변 로그
* 세션 로그
* 사용자 피드백
* 문서 인덱싱 로그
* 에러 로그
* 상담 전환 로그

먼저 제공할 뷰:

* 운영사: 헬스 대시보드, 인덱싱 현황, 장애 현황
* 고객사: 문의량, 자동응대율, 만족도, 상위 질문

이 단계에서는 **기본 지표의 신뢰도**가 중요합니다.

---

## 2단계: 품질 관리 체계 구축

목표: 답변 품질을 개선할 수 있게

추가로 쌓아야 할 것:

* 답변 근거 문서
* 검색 결과 후보군
* 신뢰도 점수
* 실패 사유 코드
* 질문 카테고리/의도 분류 결과
* 검수 라벨 데이터

추가 뷰:

* 실패 응답 분석
* 지식 공백 분석
* 고위험 질문 모니터링
* 품질 검수 큐

이 단계부터 RAG 개선이 본격적으로 가능해집니다.

---

## 3단계: 고객사 성과관리 체계 구축

목표: 챗봇이 행정업무에 얼마나 기여했는지 증명

추가로 쌓아야 할 것:

* 민원 유형 매핑
* 상담 연계 결과
* 재문의 여부
* 처리 시간 절감 추정치
* 업무시간 외 응대 데이터
* 부서/업무 단위 성과 데이터

추가 뷰:

* 민원 카테고리별 성과
* 상담 절감 효과
* 기간별 개선 추이
* 부서별 활용도

이 단계에서 고객사는 “도입 효과”를 내부 보고할 수 있습니다.

---

## 4단계: 고도화/자동화 단계

목표: 운영 공수를 줄이고 선제 대응

추가 기능:

* 이상탐지 알림
* 자동 재인덱싱
* 문서 최신화 알림
* 실패 질문 자동 클러스터링
* 고위험 응답 자동 차단
* 기관별 품질 리포트 자동 생성

추가 뷰:

* 예측형 운영 대시보드
* 개선 우선순위 추천
* 기관별 운영 자동화 현황

이 단계가 되면 운영사는 적은 인력으로 더 많은 기관을 관리할 수 있습니다.

# 8. IA 설계 원칙

이건 꼭 지켜야 합니다.

## 8-1. 운영사와 고객사 메뉴를 섞지 않기

* 운영사는 장애/품질/RAG
* 고객사는 성과/민원/개선
* 같은 데이터라도 보여주는 방식이 달라야 함

## 8-2. “지표 조회”와 “조치 액션”을 붙이기

예를 들면:

* 미응답 질문 화면에서 바로 문서 추가 요청
* 오답 화면에서 바로 FAQ 등록
* 최신화 누락 문서 화면에서 바로 재인덱싱 요청

## 8-3. 공공기관 특성상 승인 흐름 넣기

* 문서 수정 → 검수 → 승인 → 반영
* 정책 변경 → 승인 → 배포
* 이력 추적 가능해야 함

## 8-4. 모든 상세화면은 기관/부서/업무유형 필터 가능해야 함

최소 필터:

* 기관
* 기간
* 채널
* 민원 카테고리
* 응답 결과
* 문서 유형

# 9. 다음 단계로 이어질 연결 포인트

이 IA를 기준으로 다음 단계인 와이어프레임은 아래 핵심 화면부터 그리면 됩니다.

```text
1. 운영 대시보드
2. 고객사 기관 대시보드
3. 품질 검수 화면
4. RAG 인덱싱 운영 화면
5. 실패/전환 분석 화면
```

이 5개가 전체 서비스의 중심 화면입니다.

---

다음 순서로 바로 이어서 **2. 대시보드 와이어프레임** 보여드리겠습니다.

---

좋습니다. 이제 **2단계: 대시보드 와이어프레임**을 보여드리겠습니다.
여기서는 실제 서비스에서 가장 중요한 5개의 핵심 화면을 기준으로 설계합니다.

설계 기준은 다음입니다.

* 운영사는 **관제 중심**
* 고객사는 **민원성과 중심**
* QA는 **개선 액션 중심**

즉 **“보기용 대시보드”가 아니라 “조치 가능한 대시보드”**로 설계합니다.

상담사 연결 기능이 없으면, **해결율을 “직접 확정값”이 아니라 “추정 해결율” 또는 “간접 해결율”로 정의**해야 합니다.

즉,
**“사람에게 넘겼는지 여부”가 없으니, 사용자의 행동 로그로 해결 여부를 추론**하는 방식입니다.

아래처럼 단계적으로 보시면 됩니다.

---

# 1. 먼저 결론

상담사 연결이 없을 경우 해결율은 보통 3가지 방식으로 도출합니다.

1. **명시적 해결 신호**

   * 사용자가 “해결됐어요”, “도움됐어요” 같은 피드백을 남긴 경우

2. **행동 기반 간접 해결 신호**

   * 답변 후 추가 질문 없이 종료
   * 동일 질문 재발 없음
   * 관련 안내 링크/문서 클릭 후 이탈
   * 특정 목표 행동 완료

3. **복합 점수 기반 추정 해결율**

   * 여러 신호를 합쳐 “해결 가능성 점수”를 만들고 해결/미해결로 분류

실무적으로는 **1번 + 2번을 같이 쓰고**, 나중에 고도화하면서 **3번으로 발전**시키는 게 가장 좋습니다.

---

# 2. 상담사 연결이 없으면 왜 어려운가

상담사 연결이 있는 경우엔 보통 이렇게 봅니다.

* 챗봇으로 끝남 = 해결 가능성 높음
* 상담사로 넘어감 = 챗봇 미해결

그런데 상담사 연결이 없으면,
사용자가 그냥 나가버린 건지
정말 해결된 건지
불만족해서 포기한 건지 구분이 안 됩니다.

그래서 해결율을 아래처럼 나눠야 합니다.

* **확정 해결율**
* **추정 해결율**
* **미해결 의심률**

즉, 하나의 숫자로만 보지 않는 게 중요합니다.

---

# 3. 가장 기본적인 방법: 명시적 피드백 기반 해결율

가장 정확한 방식은 답변 후 사용자에게 간단히 묻는 것입니다.

예:

* 도움이 되었나요?
* 궁금한 점이 해결되었나요?
* 원하는 정보를 찾으셨나요?

버튼 예:

* 해결됐어요
* 아직 잘 모르겠어요
* 추가로 궁금해요

이 경우 공식은 단순합니다.

```text
명시적 해결율 = "해결됐어요" 응답 수 / 피드백 응답 수
```

예:

```text
피드백 응답 1,000건
- 해결됐어요: 620
- 아직 잘 모르겠어요: 250
- 추가로 궁금해요: 130

명시적 해결율 = 620 / 1,000 = 62%
```

주의할 점은,
전체 사용자가 피드백을 누르지 않기 때문에 이 값만으로 전체 해결율로 보면 안 됩니다.

그래서 보통 이렇게 같이 봅니다.

```text
명시적 해결율
피드백 응답률
```

예:

```text
전체 세션 10,000건
피드백 응답 1,000건
해결됐어요 620건

명시적 해결율: 62%
피드백 응답률: 10%
```

즉 “응답한 사용자 기준 62%”라고 해석해야 합니다.

---

# 4. 피드백이 없을 때: 행동 기반 해결 추정

이게 실제로 더 많이 쓰입니다.

상담 연결이 없으면 아래 신호를 조합해 해결 여부를 추정합니다.

## 4-1. 추가 질문 없이 종료

예:

* 질문 1개 또는 2개 내로 종료
* 마지막 답변 이후 일정 시간 내 재질문 없음

이 경우 해결 가능성이 높다고 볼 수 있습니다.

하지만 단점도 있습니다.

* 실망해서 그냥 나간 것일 수도 있음

그래서 이것만 단독 지표로 쓰면 위험합니다.

---

## 4-2. 동일/유사 질문 재발 없음

예:

* 사용자가 7일 이내 같은 질문을 다시 하지 않음
* 같은 민원 카테고리 질문 재유입이 없음

이 경우 해결됐을 가능성이 높습니다.

공식 예:

```text
재문의 없는 세션 비율 = 특정 질문 후 7일 내 유사 재문의가 없는 세션 / 전체 세션
```

이건 꽤 유용합니다.

---

## 4-3. 안내 후 목표 행동 완료

공공기관 챗봇이면 특히 중요합니다.

예:

* 민원 신청 페이지 이동
* 서식 다운로드
* 공고문 열람
* 담당 부서 연락처 조회
* 온라인 접수 페이지 진입
* 증명서 발급 페이지 클릭

즉 챗봇의 목적이 “답변”이 아니라 “업무 완료 유도”라면,
그 행동이 일어나면 해결 가능성이 높다고 봅니다.

예시 공식:

```text
행동 완료 기반 해결율 = 목표 행동 완료 세션 수 / 해당 유형 전체 세션 수
```

예:

```text
주민등록등본 발급 문의 1,000건
발급 페이지 클릭 710건

행동 완료 기반 해결율 = 71%
```

물론 클릭만 했다고 진짜 해결된 건 아닐 수 있으니
정확히는 “업무진행 유도율”에 가깝습니다.

---

## 4-4. 답변 후 문서/링크 열람 + 종료

예:

* 챗봇이 관련 문서를 제시
* 사용자가 문서를 클릭
* 이후 추가 질문 없이 종료

이건 “정보 획득 후 종료”로 볼 수 있어서 해결 신호로 활용 가능합니다.

---

# 5. 가장 현실적인 방식: 추정 해결율 모델

실무적으로는 아래처럼 **복합 조건**으로 추정 해결율을 만듭니다.

예시 규칙:

```text
해결 추정 세션 조건
- 미응답/오답 플래그 없음
- 마지막 답변에 출처 포함
- 답변 후 30초 이상 체류 또는 링크 클릭
- 10분 내 추가 질문 없음
- 7일 내 동일/유사 재문의 없음
```

이 조건을 만족하면 “해결 추정”으로 분류합니다.

공식:

```text
추정 해결율 = 해결 추정 세션 수 / 전체 세션 수
```

예:

```text
전체 세션 10,000
해결 추정 세션 6,800

추정 해결율 = 68%
```

이건 실제 운영에서 가장 많이 쓰기 좋은 방식입니다.

---

# 6. 공공기관 챗봇에 맞는 해결율 정의 방법

공공기관용은 일반 CS 챗봇과 조금 다르게 잡아야 합니다.

왜냐하면 공공기관 민원은 크게 3가지로 나뉩니다.

1. **단순 정보 안내형**

   * 운영시간, 위치, 제출서류, 절차 등

2. **업무 유도형**

   * 신청 페이지 이동, 서식 다운로드, 접수 방법 안내

3. **판단/심사 필요형**

   * 자격 여부, 예외 적용, 개별 사정 판단 등

이 3개를 같은 기준으로 보면 안 됩니다.

---

## 6-1. 단순 정보 안내형

해결 판단 기준:

* 피드백 긍정
* 추가 질문 없음
* 재문의 없음

## 6-2. 업무 유도형

해결 판단 기준:

* 링크 클릭
* 신청/다운로드/조회 페이지 이동
* 후속 행동 완료

## 6-3. 판단 필요형

이건 챗봇이 완결 해결하는 게 아니라
**적절한 안내를 했는지**를 봐야 합니다.

해결율 대신 이런 지표가 더 적절합니다.

* 적정 안내율
* 담당부서 안내 성공률
* 민감 질문 안전응답률

즉, 모든 질문을 “해결/미해결”로 억지로 나누면 안 됩니다.

---

# 7. 그래서 추천하는 KPI 체계

상담사 연결이 없을 때는 해결율을 하나만 두지 말고 아래처럼 가져가는 게 좋습니다.

## 필수 KPI 1: 명시적 해결율

```text
"해결됐어요" 응답 수 / 피드백 응답 수
```

## 필수 KPI 2: 추정 해결율

```text
행동 로그 기준 해결 추정 세션 수 / 전체 세션 수
```

## 필수 KPI 3: 재문의율

```text
동일/유사 질문이 일정 기간 내 다시 들어온 비율
```

## 필수 KPI 4: 업무 유도 성공률

```text
신청/조회/다운로드/민원접수 등 목표 행동 완료 비율
```

## 필수 KPI 5: 미해결 의심률

```text
저만족 + 재문의 + 반복질문 + 장시간 대화 후 종료 세션 비율
```

이렇게 해야 숫자를 균형 있게 볼 수 있습니다.

# 8. 가장 추천하는 계산 구조

제가 추천하는 건 아래 방식입니다.

## A. 세션 상태를 4가지로 분류

```text
1. 해결 확정
- 사용자가 해결됐다고 응답

2. 해결 추정
- 피드백은 없지만 행동 신호가 긍정적

3. 미해결 의심
- 반복 질문, 저만족, 재문의, 장시간 반복

4. 판단 불가
- 데이터 부족
```

## B. 대시보드에는 이렇게 노출

```text
전체 세션 100,000건

해결 확정      18%
해결 추정      46%
미해결 의심    21%
판단 불가      15%
```

이 구조가 가장 정직하고 운영에도 유용합니다.

# 11. 한 줄 정리

상담사 연결이 없을 경우 해결율은
**“사람에게 안 넘겼는지”로 보는 게 아니라, “사용자의 피드백과 후속 행동으로 해결 가능성을 추정하는 방식”**으로 도출해야 합니다.

즉 실무적으로는:

```text
해결율 = 명시적 해결율 + 행동 기반 추정 해결율
보조지표 = 재문의율, 목표행동 성공률, 미해결 의심률
```

이렇게 가는 게 가장 맞습니다.

원하시면 다음으로 바로
**“상담사 연결 없는 공공기관 챗봇용 해결율 KPI 정의서”** 형태로 표준안까지 만들어드리겠습니다.

---

좋습니다.
**공공기관용 챗봇 + RAG + 어드민 분석**을 제대로 만들려면 DB 구조를 먼저 안정적으로 설계해야 합니다.

핵심 원칙은 다음입니다.

1. **로그 중심 구조 (event driven)**
2. **세션 → 메시지 → 행동 로그 분리**
3. **RAG 파이프라인 로그 별도 관리**
4. **품질/검수 데이터 별도 레이어**
5. **기관 멀티테넌트 구조**

아래는 **실제 서비스 가능한 수준의 테이블 구조**입니다.

---

# 1. 멀티테넌트 기본 구조

공공기관 플랫폼이므로 **기관 단위 isolation**이 필요합니다.

## organizations

```sql
organizations
------------
id (PK)
name
type                -- government / agency / public_org
region
created_at
status
```

---

## services (기관별 챗봇)

```sql
services
------------
id (PK)
organization_id (FK)
name
channel_type        -- web / mobile / kakao / api
status
created_at
```

# 다음 단계

다음 단계로 가면 **훨씬 중요한 부분**이 나옵니다.

**"이 DB에서 실제 KPI를 어떻게 계산할 것인가"**

예:

* 상담 필요 질문

원하시면 다음으로
**"공공기관 챗봇 KPI 계산 로직 + SQL 수준 설계"**까지 이어서 설명해 드리겠습니다.

---

좋습니다.
그럼 이제 위 DB 구조를 기준으로 **실제 KPI를 어떻게 계산할지**를 잡아보겠습니다.

이번 단계는 단순 설명이 아니라,
**어떤 테이블에서 어떤 값을 뽑아 어떤 공식으로 계산하는지**까지 연결해서 보겠습니다.

흐름은 아래 순서로 가면 됩니다.

1. KPI 분류
2. KPI별 계산 기준
3. SQL 수준 예시
4. 주의할 점
5. 추천 집계 구조

# 2. 이용 KPI

이건 가장 기본입니다.

## 2-1. 총 세션 수

기준:

* 특정 기간 내 생성된 세션 수

```sql
SELECT COUNT(*) AS total_sessions
FROM chat_sessions
WHERE organization_id = :org_id
  AND session_start_at >= :start_date
  AND session_start_at < :end_date;
```

---

## 2-2. 총 질문 수

기준:

* user role 메시지 수
* 또는 questions 테이블 기준 count

questions 테이블 기준이 더 안정적입니다.

```sql
SELECT COUNT(*) AS total_questions
FROM questions q
JOIN chat_sessions s ON s.id = q.session_id
WHERE s.organization_id = :org_id
  AND q.created_at >= :start_date
  AND q.created_at < :end_date;
```

---

## 2-3. 순 사용자 수(UV)

익명 사용자 기준입니다.

```sql
SELECT COUNT(DISTINCT s.user_id) AS unique_users
FROM chat_sessions s
WHERE s.organization_id = :org_id
  AND s.session_start_at >= :start_date
  AND s.session_start_at < :end_date;
```

익명 ID 기반이면 `user_id` 대신 `anonymous_id` 기준으로 집계해도 됩니다.

---

## 2-4. 세션당 평균 질문 수

```sql
SELECT ROUND(AVG(question_count), 2) AS avg_questions_per_session
FROM chat_sessions
WHERE organization_id = :org_id
  AND session_start_at >= :start_date
  AND session_start_at < :end_date;
```

또는 실측 기준:

```sql
SELECT ROUND(COUNT(q.id) * 1.0 / COUNT(DISTINCT s.id), 2) AS avg_questions_per_session
FROM chat_sessions s
LEFT JOIN questions q ON q.session_id = s.id
WHERE s.organization_id = :org_id
  AND s.session_start_at >= :start_date
  AND s.session_start_at < :end_date;
```

# 3. 해결/성과 KPI

여기가 핵심입니다.
앞서 말씀드린 것처럼 상담사 연결이 없으면 **확정 해결율 + 추정 해결율**로 나눠야 합니다.

---

## 3-1. 명시적 해결율

기준:

* feedbacks.feedback_type = 'solved'

```sql
SELECT
  COUNT(CASE WHEN feedback_type = 'solved' THEN 1 END) * 100.0
  / NULLIF(COUNT(*), 0) AS explicit_resolution_rate
FROM feedbacks f
JOIN chat_sessions s ON s.id = f.session_id
WHERE s.organization_id = :org_id
  AND f.created_at >= :start_date
  AND f.created_at < :end_date;
```

이 값은 반드시
**“피드백 응답자 기준”**이라고 명시해야 합니다.

---

## 3-2. 피드백 응답률

이것도 같이 봐야 합니다.

```sql
SELECT
  COUNT(DISTINCT f.session_id) * 100.0
  / NULLIF(COUNT(DISTINCT s.id), 0) AS feedback_response_rate
FROM chat_sessions s
LEFT JOIN feedbacks f ON f.session_id = s.id
WHERE s.organization_id = :org_id
  AND s.session_start_at >= :start_date
  AND s.session_start_at < :end_date;
```

---

## 3-3. 추정 해결율

예시 규칙:

* 세션 내 미응답 없음
* 부정 피드백 없음
* 링크 클릭 또는 문서 열람 있음
  또는
* 마지막 답변 이후 추가 질문 없음
* 7일 내 동일 주제 재문의 없음

이런 기준으로 `resolved_flag`를 배치로 계산해두는 걸 권장합니다.

예:

```sql
SELECT
  COUNT(CASE WHEN resolved_flag = true THEN 1 END) * 100.0
  / NULLIF(COUNT(*), 0) AS estimated_resolution_rate
FROM chat_sessions
WHERE organization_id = :org_id
  AND session_start_at >= :start_date
  AND session_start_at < :end_date;
```

실무적으로는 `resolved_flag`를 실시간 계산보다
**배치/집계 로직으로 갱신**하는 게 좋습니다.

---

## 3-4. 미해결 의심률

예시 기준:

* unsolved 피드백
* fallback 질문 포함
* 동일/유사 질문 재문의
* 세션 길이 길지만 목표행동 없음

이를 세션 플래그로 저장:

```sql
chat_sessions.suspected_unresolved_flag
```

계산:

```sql
SELECT
  COUNT(CASE WHEN suspected_unresolved_flag = true THEN 1 END) * 100.0
  / NULLIF(COUNT(*), 0) AS suspected_unresolved_rate
FROM chat_sessions
WHERE organization_id = :org_id
  AND session_start_at >= :start_date
  AND session_start_at < :end_date;
```

---

## 3-5. 재문의율

기준:

* `revisit_within_7d = true`

```sql
SELECT
  COUNT(CASE WHEN revisit_within_7d = true THEN 1 END) * 100.0
  / NULLIF(COUNT(*), 0) AS revisit_rate_7d
FROM chat_sessions
WHERE organization_id = :org_id
  AND session_start_at >= :start_date
  AND session_start_at < :end_date;
```

주의:
이건 실제로는 “동일/유사 주제 재문의율”로 보는 게 더 좋습니다.
그럼 session 단위보다 **question_topic 기준**으로 봐야 합니다.

---

## 3-6. 목표 행동 성공률

공공기관용에서 매우 중요합니다.

예:

* 민원 접수 페이지 클릭
* 서식 다운로드
* 증명서 발급 페이지 진입

`user_actions.action_type = 'target_action'`
`action_value = 'completed'`

```sql
SELECT
  COUNT(CASE WHEN ua.action_type = 'target_action'
              AND ua.action_value = 'completed' THEN 1 END) * 100.0
  / NULLIF(COUNT(DISTINCT q.id), 0) AS target_action_success_rate
FROM questions q
JOIN chat_sessions s ON s.id = q.session_id
LEFT JOIN user_actions ua ON ua.question_id = q.id
WHERE s.organization_id = :org_id
  AND q.created_at >= :start_date
  AND q.created_at < :end_date
  AND q.question_category = :category;
```

이건 카테고리별로 보는 게 훨씬 의미 있습니다.

## 4-2. 오답 의심률

기준:

* qa_reviews.review_type = 'wrong_answer'

```sql
SELECT
  COUNT(DISTINCT qr.question_id) * 100.0
  / NULLIF(COUNT(DISTINCT q.id), 0) AS suspected_wrong_answer_rate
FROM questions q
JOIN chat_sessions s ON s.id = q.session_id
LEFT JOIN qa_reviews qr
  ON qr.question_id = q.id
 AND qr.review_type = 'wrong_answer'
WHERE s.organization_id = :org_id
  AND q.created_at >= :start_date
  AND q.created_at < :end_date;
```

## 4-3. 사용자 만족도

rating 1~5 기준

```sql
SELECT ROUND(AVG(rating), 2) AS avg_satisfaction
FROM feedbacks f
JOIN chat_sessions s ON s.id = f.session_id
WHERE s.organization_id = :org_id
  AND f.created_at >= :start_date
  AND f.created_at < :end_date

```

---

## 4-4. 출처 포함 답변 비율

이건 answers에 컬럼을 두는 걸 추천합니다.

예:

```text
answers.has_citation boolean
```

```sql
SELECT
  COUNT(CASE WHEN a.has_citation = true THEN 1 END) * 100.0
  / NULLIF(COUNT(*), 0) AS citation_coverage_rate
FROM answers a
JOIN questions q ON q.id = a.question_id
JOIN chat_sessions s ON s.id = q.session_id
WHERE s.organization_id = :org_id
  AND a.created_at >= :start_date
  AND a.created_at < :end_date;
```

## 4-5. 평균 응답 시간

```sql
SELECT ROUND(AVG(a.response_time_ms), 2) AS avg_response_time_ms
FROM answers a
JOIN questions q ON q.id = a.question_id
JOIN chat_sessions s ON s.id = q.session_id
WHERE s.organization_id = :org_id
  AND a.created_at >= :start_date
  AND a.created_at < :end_date;
```

퍼센타일도 같이 보는 게 좋습니다.

```sql
SELECT
  PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY a.response_time_ms) AS p50,
  PERCENTILE_CONT(0.9) WITHIN GROUP (ORDER BY a.response_time_ms) AS p90,
  PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY a.response_time_ms) AS p95
FROM answers a
JOIN questions q ON q.id = a.question_id
JOIN chat_sessions s ON s.id = q.session_id
WHERE s.organization_id = :org_id
  AND a.created_at >= :start_date
  AND a.created_at < :end_date;
```

# 5. 민원/업무 분석 KPI

---

## 5-1. 민원 유형별 질문 수

```sql
SELECT
  q.question_category,
  COUNT(*) AS question_count
FROM questions q
JOIN chat_sessions s ON s.id = q.session_id
WHERE s.organization_id = :org_id
  AND q.created_at >= :start_date
  AND q.created_at < :end_date
GROUP BY q.question_category
ORDER BY question_count DESC;
```

---

## 5-2. 민원 유형별 해결율

```sql
SELECT
  q.question_category,
  COUNT(CASE WHEN s.resolved_flag = true THEN 1 END) * 100.0
  / NULLIF(COUNT(*), 0) AS resolution_rate
FROM questions q
JOIN chat_sessions s ON s.id = q.session_id
WHERE s.organization_id = :org_id
  AND q.created_at >= :start_date
  AND q.created_at < :end_date
GROUP BY q.question_category
ORDER BY resolution_rate DESC;
```

세션 기준인지 질문 기준인지 정의를 명확히 해야 합니다.
공공기관에선 보통 **질문 기준 + 세션 기준 둘 다** 보는 게 좋습니다.

---

## 5-3. 자주 묻는 질문 TOP N

```sql
SELECT
  q.normalized_question,
  COUNT(*) AS cnt
FROM questions q
JOIN chat_sessions s ON s.id = q.session_id
WHERE s.organization_id = :org_id
  AND q.created_at >= :start_date
  AND q.created_at < :end_date
GROUP BY q.normalized_question
ORDER BY cnt DESC
LIMIT 20;
```

---

## 5-4. 반복 민원 TOP N

question_clusters 활용:

```sql
SELECT
  qc.cluster_label,
  COUNT(*) AS question_count
FROM question_cluster_map qcm
JOIN question_clusters qc ON qc.id = qcm.cluster_id
JOIN questions q ON q.id = qcm.question_id
JOIN chat_sessions s ON s.id = q.session_id
WHERE s.organization_id = :org_id
  AND q.created_at >= :start_date
  AND q.created_at < :end_date
GROUP BY qc.cluster_label
ORDER BY question_count DESC
LIMIT 20;
```

# 6. RAG 운영 KPI

## 6-1. 검색 성공률

`rag_search_logs.search_success_flag`

```sql
SELECT
  COUNT(CASE WHEN rsl.search_success_flag = true THEN 1 END) * 100.0
  / NULLIF(COUNT(*), 0) AS search_success_rate
FROM rag_search_logs rsl
JOIN questions q ON q.id = rsl.question_id
JOIN chat_sessions s ON s.id = q.session_id
WHERE s.organization_id = :org_id
  AND rsl.created_at >= :start_date
  AND rsl.created_at < :end_date;
```

## 6-2. 검색 무결과율

컬럼 예:

```text
rag_search_logs.result_count
```

```sql
SELECT
  COUNT(CASE WHEN rsl.result_count = 0 THEN 1 END) * 100.0
  / NULLIF(COUNT(*), 0) AS zero_result_rate
FROM rag_search_logs rsl
JOIN questions q ON q.id = rsl.question_id
JOIN chat_sessions s ON s.id = q.session_id
WHERE s.organization_id = :org_id
  AND rsl.created_at >= :start_date
  AND rsl.created_at < :end_date;
```

## 6-3. 문서 참조 빈도

질문별 top document 연결 테이블이 있으면 가장 좋습니다.
현재 구조라면 `retrieved_document_ids`를 정규화하는 게 더 좋습니다.

추천 추가 테이블:

```sql
rag_retrieved_documents
-----------------------
id
question_id
document_id
chunk_id
rank_order
score
selected_for_answer
created_at
```

그럼 문서 참조 빈도는 이렇게 됩니다.

```sql
SELECT
  d.id,
  d.title,
  COUNT(*) AS ref_count
FROM rag_retrieved_documents rrd
JOIN documents d ON d.id = rrd.document_id
JOIN questions q ON q.id = rrd.question_id
JOIN chat_sessions s ON s.id = q.session_id
WHERE s.organization_id = :org_id
  AND rrd.created_at >= :start_date
  AND rrd.created_at < :end_date
  AND rrd.selected_for_answer = true
GROUP BY d.id, d.title
ORDER BY ref_count DESC
LIMIT 20;
```

# 10. KPI 설계 시 가장 중요한 주의점

# 다음 단계 (진짜 중요한 것)

지금까지는 **기획 + 데이터 설계**였습니다.

다음 단계로 가면 **아키텍처 레벨 설계**가 나옵니다.

특히 공공기관용 RAG 챗봇에서 중요한 것:

1️⃣ **RAG 파이프라인 아키텍처**
2️⃣ **문서 업데이트 자동화 구조**
3️⃣ **LLM 프롬프트 운영 관리 시스템**
4️⃣ **멀티기관 챗봇 아키텍처**

이건 실제 **제품 수준 설계**입니다.

원하시면 다음으로
**“공공기관 RAG 챗봇 전체 아키텍처 (운영 가능한 수준)”**을 이어서 설명드리겠습니다.

---

좋습니다. 그럼 이 전제를 반영해서 설계를 한 단계 더 현실적으로 바꿔야 합니다.

지금 말씀하신 문서 소스는 단순 업로드 문서가 아니라 아래처럼 **성격이 완전히 다른 지식원**입니다.

* 홈페이지 크롤링 데이터
* 법령/조례/행정규칙
* 민원편람
* 실무편람
* 공지사항/보도자료/고시
* FAQ/담당부서 안내
* 내부 운영 매뉴얼

이 경우 핵심은 **문서를 한 테이블에만 넣는 것**이 아니라,
**“문서 원천(source) / 문서 유형(type) / 최신성(version) / 신뢰도(priority)”**를 분리해서 관리하는 것입니다.

즉 공공기관 RAG는 그냥 “문서 검색”이 아니라
**출처 우선순위가 있는 행정지식 검색 시스템**으로 가야 합니다.

# 1. 왜 이 구분이 중요한가

예를 들어 같은 질문이라도 답변에 써야 하는 근거가 다릅니다.

### 예시 1. 법적 기준 질문

“장애인 주차구역 위반 과태료 얼마인가요?”

이건 우선순위가 보통 이렇게 가야 합니다.

```text
법령/조례 > 민원편람 > 홈페이지 안내문 > FAQ
```

### 예시 2. 신청 절차 질문

“주민등록등본 어디서 발급하나요?”

이건 보통 이렇게 갑니다.

```text
민원편람 > 홈페이지 민원안내 > FAQ > 실무편람
```

### 예시 3. 내부 처리 질문

“담당 공무원이 민원 접수 후 어떤 절차로 검토하나요?”

이건 외부 민원인용이 아니라 내부용일 가능성이 높아서

```text
실무편람 > 내부 매뉴얼 > 관련 법령
```

즉 문서를 다 같이 벡터DB에 넣기만 하면 안 되고,
**질문 유형에 따라 참조 우선순위를 조절**해야 합니다.

# 13. 한 줄로 정리

지금 주신 조건을 반영하면 이 시스템은 더 이상 단순한 “문서형 RAG”가 아니라,

**“홈페이지 크롤링 + 법령 + 민원편람 + 실무편람을 함께 관리하는 공공기관 지식운영 플랫폼”**으로 설계해야 합니다.

그래서 DB도 반드시 아래를 구분해야 합니다.

```text
1. 문서 원천
2. 문서 유형
3. 공개 범위
4. 최신성/버전
5. 출처 우선순위
6. 검색/답변 사용 이력
```

이 6개가 빠지면 운영하면서 반드시 꼬입니다.

---

다음으로 이어서 가장 자연스러운 건
**이 구조를 반영한 최종 ERD 초안**을 텍스트로 정리해드리는 것입니다.

---

좋습니다. 이제까지 나온 내용을 **하나의 최종 ERD(엔티티 관계 구조)**로 정리하겠습니다.
목표는 **“공공기관 RAG 챗봇 운영 플랫폼”을 실제로 구현 가능한 수준의 데이터 구조**로 만드는 것입니다.

설계 기준은 다음 5가지입니다.

* 멀티기관 운영 (multi-tenant)
* 다양한 문서 소스 관리 (crawl / 법령 / 민원편람 / 실무편람)
* RAG 파이프라인 추적 가능
* QA / 품질 개선 가능
* KPI / 어드민 분석 가능

---

# 마지막으로 중요한 질문 하나

이 플랫폼을 어디까지 확장할지에 따라 **아키텍처가 완전히 달라집니다.**

특히 다음 두 가지가 중요합니다.

1️⃣ **멀티기관 SaaS 플랫폼인지**
(여러 지자체 / 기관이 동시에 사용하는 구조)

2️⃣ **LLM을 기관별로 커스터마이징할지**
(프롬프트 / 정책 / 검색 전략)

이 두 가지에 따라

* DB 분리 전략
* vectorDB 구조
* RAG 정책
* 프롬프트 관리

가 크게 달라집니다.

원하시면 다음으로
**“공공기관 RAG 챗봇 전체 시스템 아키텍처 (Infra + VectorDB + Pipeline)”**를 이어서 설계해드리겠습니다.

---

좋습니다. 그럼 이제 **공공기관용 RAG 챗봇 전체 시스템 아키텍처**를 운영 가능한 수준으로 이어서 설계하겠습니다.

이번 단계는 DB보다 한 단계 위입니다.
즉, **어떤 컴포넌트를 두고, 문서가 어떻게 들어오고, 질문이 들어왔을 때 어떤 순서로 처리하며, 운영사는 어디를 보고 관리하는지**까지 한 번에 정리하겠습니다.

핵심 전제는 이미 분명합니다.

* 멀티기관 SaaS 구조
* 문서 소스가 다양함
  (홈페이지 크롤링, 법령, 민원편람, 실무편람, FAQ, 공지)
* 공공기관 특성상 **출처, 최신성, 공개범위, 감사추적**이 중요함
* 고객사에는 민원 성과가 보여야 하고
* 운영사에는 장애/품질/RAG 운영 상태가 보여야 함

또한 검색은 **BM25 같은 키워드 검색과 벡터 검색을 결합한 하이브리드 검색**이 공공기관 질의에 유리하고, 멀티테넌시 관점에서는 벡터 저장소를 tenant metadata로 분리하는 접근이 일반적입니다. OpenSearch는 하이브리드 검색을 공식적으로 지원하고, Qdrant는 payload 기반 멀티테넌시와 tenant indexing을 권장합니다. ([OpenSearch Documentation][1])

# 2. 레이어별 역할

## 2-1. Channel Layer

여기는 실제 사용자가 접속하는 채널입니다.

* 기관 홈페이지 챗봇 위젯
* 모바일 웹/앱
* 공공기관 내부 업무포털
* 외부 메신저 연동
* API 연동 채널

핵심 포인트:

* **채널은 여러 개여도, 질문 처리 엔진은 하나**여야 운영이 쉽습니다.
* 채널별 UI만 다르고, 세션/질문/로그 구조는 공통으로 가져가야 합니다.

---

## 2-2. Gateway / App Layer

여기는 모든 요청이 들어오는 입구입니다.

주요 기능:

* 인증/인가
* 기관 식별
* Rate limiting
* IP / bot filtering
* request validation
* channel metadata 부착

예:

* 어떤 기관의 어떤 서비스인지
* 시민용인지 내부직원용인지
* 어떤 채널에서 들어온 질문인지

이 레이어에서 `organization_id`, `service_id`, `audience_type`가 결정됩니다.

---

## 2-3. Chat Orchestration Layer

이 레이어가 사실상 제품의 두뇌입니다.

역할:

* 질문 전처리
* 카테고리/의도 분류
* 민감 질문 판별
* 검색 정책 선택
* 답변 생성 전략 선택
* fallback / 안내 문구 결정
* 로그 적재

즉, LLM을 바로 부르는 게 아니라 **오케스트레이터가 먼저 판단**해야 합니다.

예:

* 법령형 질문 → 법령 우선 검색
* 민원 신청 질문 → 민원편람 + 홈페이지 절차 우선
* 내부업무 질문 → 실무편람 허용
* 시민 채널 → internal/restricted 문서 차단

---

## 2-4. Retrieval / RAG Layer

여기가 검색/근거 생성 레이어입니다.

구성:

* query rewrite
* hybrid retrieval
* metadata filtering
* rerank
* citation selection
* grounding check

공공기관에서는 특히 아래가 중요합니다.

* 하이브리드 검색
* 최신 문서 우선
* 공개범위 필터
* 출처 유형별 우선순위
* 법령/민원편람/홈페이지의 역할 분리

OpenSearch는 BM25와 semantic/vector를 결합하는 하이브리드 검색을 지원하고, score normalization과 결합 방식을 제공해 검색 정밀도와 재현율을 함께 개선하는 데 초점을 둡니다. ([OpenSearch Documentation][1])

---

## 2-5. Knowledge Ingestion Layer

문서가 들어오는 쪽입니다.

입력 소스:

* 홈페이지 크롤링
* 법령/조례 동기화
* 민원편람 배치 수집
* 실무편람 업로드
* FAQ 수기 등록
* 공지/보도자료 수집

기능:

* 수집
* 파싱
* 정제
* 문서 분류
* chunking
* embedding
* indexing
* 변경 감지
* 버전 갱신

이 레이어가 안정적이지 않으면 RAG 품질이 계속 흔들립니다.

---

## 2-6. Data / Analytics Layer

여기는 운영과 리포팅용입니다.

저장소를 분리해서 보는 걸 권장합니다.

* **OLTP DB**: 세션, 질문, 문서 메타, QA
* **Search Index**: BM25/필터/하이브리드
* **Vector DB**: semantic retrieval
* **Object Storage**: 원문/첨부
* **Analytics Store**: 대시보드용 집계

즉, 하나의 DB로 다 하려 하지 말고 역할을 나누는 게 좋습니다.

---

## 2-7. Admin / QA Layer

여기는 운영사가 매일 쓰는 실무 화면입니다.

* 운영 대시보드
* 기관별 상태
* 인덱싱/크롤링 상태
* 실패 질문 검수
* 문서 최신화 관리
* QA 라벨링
* 출처 누락/내부문서 노출 감시
* 리포트 다운로드

* 크롤링 원문
* 파싱 산출물
* 첨부 이미지
* 리포트 파일

이유:

* 원문 보관/감사추적
* 재파싱 가능
* 버전 이력 보존

---

## Search Engine

추천 용도:

* BM25 검색
* 필터 검색
* 하이브리드 검색
* faceted analytics 일부

공공기관에서는 키워드가 매우 중요합니다.
예:

* 법령명
* 조문 번호
* 민원명
* 기관명

그래서 벡터만으로는 부족합니다.

## Vector DB

추천 용도:

* semantic retrieval
* rerank 후보 확보
* 긴 문장 의미 검색

멀티기관 SaaS라면 기관별 컬렉션을 무작정 나누기보다, tenant metadata와 payload filtering 기반 멀티테넌시가 일반적으로 더 효율적입니다. Qdrant 공식 문서도 대부분의 경우 단일 collection + payload partitioning을 권장하고, tenant index를 통해 테넌트별 서브인덱스를 만들 수 있다고 설명합니다. ([Qdrant][2])

## Analytics DB

추천 용도:

* 일/주/월 집계
* 기관별 KPI
* 카테고리별 KPI
* 운영 리포트

실시간 원천 로그로 대시보드 그리면 느려집니다.
그래서 배치 집계 레이어가 필요합니다.

# 5. 실전 추천 스택

특정 벤더 종속 없이 보면 아래 조합이 현실적입니다.

## 권장안 A: 안정형

* PostgreSQL
* OpenSearch
* Qdrant
* S3 호환 Object Storage
* Redis
* Queue (RabbitMQ / Kafka / SQS 계열)

이 조합의 장점:

* 키워드 + 벡터를 분리해 운영 가능
* 공공기관형 필터/검색/감사 대응에 강함
* 운영 화면 설계가 쉬움

## 권장안 B: 단순형

* PostgreSQL
* pgvector
* Object Storage
* Redis

이건 초기에 빠르게 만들기 좋지만,
기관 수와 문서량이 커지면 검색/필터/운영성이 약해질 수 있습니다.

즉,

* **PoC / 초기 구축**: 단순형 가능
* **여러 기관 동시 운영 SaaS**: 안정형 권장

# 6. 문서 수집 파이프라인 아키텍처

문서 소스가 다양한 만큼 ingestion도 분리해야 합니다.

## 6-1. 크롤링 파이프라인

흐름:

```text
스케줄러
→ 기관별 크롤링 잡 생성
→ fetch
→ HTML 본문 추출
→ 메뉴 경로 추출
→ boilerplate 제거
→ 문서 분류
→ hash 비교
→ 변경 시 새 버전 생성
→ chunk / embed / index
```

핵심:

* 본문 추출 품질
* 중복 제거
* 페이지 변경 감지
* 깨진 페이지/404 감시

---

## 6-2. 법령 동기화 파이프라인

흐름:

```text
법령 source sync
→ 문서 메타 수집
→ 조문 단위 파싱
→ 개정 여부 비교
→ 현행/구버전 분리
→ article chunk 생성
→ embed / index
```

핵심:

* 조문 단위 관리
* 시행일 기준 버전 관리
* 현행 법령 우선 사용

---

## 6-3. 민원편람 파이프라인

흐름:

```text
수집
→ 템플릿 구조 인식
→ 서비스명 / 처리기간 / 수수료 / 필요서류 추출
→ 구조화 필드 저장
→ 설명 텍스트 chunk 생성
→ index
```

핵심:

* 정형 데이터화
* 절차형 chunk
* 민원명 표준화

---

## 6-4. 실무편람/내부문서 파이프라인

흐름:

```text
업로드
→ 보안등급 태깅
→ 공개범위 판정
→ 내부 전용 여부 설정
→ chunk / embed / index
→ 시민 채널 차단 규칙 반영
```

핵심:

* 내부문서가 시민 답변에 노출되지 않도록 강한 필터 필요

# 다음 단계 (중요)

지금까지는 **플랫폼 아키텍처**였습니다.

하지만 실제 제품을 만들 때 가장 중요한 것은 다음입니다.

**어드민 화면 설계**

즉

```text
운영사 Admin
고객사 Admin
QA Admin
```

이 화면이 어떻게 구성되느냐가
챗봇 품질을 결정합니다.

---

원하시면 다음 단계로 **공공기관 RAG 챗봇 Admin UI 전체 설계 (실제 SaaS 수준)**를 보여드리겠습니다.

여기서 보통 사람들이 놀라는 부분이 하나 있습니다.

**“RAG 시스템의 품질은 LLM보다 Admin UI에서 결정된다”**

왜 그런지도 같이 설명해 드리겠습니다.

---

좋습니다.
그럼 이제 **공공기관 RAG 챗봇 Admin UI 전체 설계**로 들어가겠습니다.

말씀드린 것처럼, 이 플랫폼의 품질은 LLM 자체보다 **운영 화면이 얼마나 잘 설계되어 있는지**에 크게 좌우됩니다.

이유는 단순합니다.

* 오답을 발견하는 곳도 어드민
* 문서를 보강하는 곳도 어드민
* 크롤링 실패를 잡는 곳도 어드민
* 기관별 성과를 확인하는 곳도 어드민
* 정책을 조정하는 곳도 어드민

즉, **RAG는 운영형 제품**이라서
좋은 모델보다 **좋은 운영도구**가 더 중요할 때가 많습니다.

# 2. 운영사 Admin

운영사 Admin은 “예쁜 통계 화면”이 아니라
**관제실 + 운영실 + 품질센터**처럼 설계해야 합니다.

핵심 목표는 4가지입니다.

```text
1. 기관 전체 상태 한눈에 파악
2. 장애/품질 이상 조기 탐지
3. 문서/RAG 파이프라인 관리
4. 운영 효율 극대화
```

## 2-2. 운영 대시보드

첫 화면은 “오늘 지금 무슨 일이 벌어지고 있는가”가 보여야 합니다.

### 상단 KPI 카드

```text
- 운영 중 기관 수

- 오늘 총 질문 수
- 평균 응답시간
- fallback rate
- error rate
- citation 누락 비율
- 내부문서 참조 위험 건수
```

### 중앙 핵심 영역

```text
A. 기관 헬스맵
B. 이상 탐지 피드
C. ingestion 파이프라인 상태
D. 최근 주요 장애/이슈
```

### 하단 영역

```text
- 질문량 추이
- 기관별 품질 순위
- 기관별 RAG 실패율
- 비용 추이
```

## 2-5. 문서 소스 관리 화면

```text

├─ 법령/조례
├─ 민원편람
├─ 실무편람
├─ FAQ/공지
└─ 업로드 문서
```

### 각 탭에서 공통으로 보여줄 컬럼

```text
- 문서명

- 출처
- 버전
- 공개범위
- 최신 여부
- 마지막 수집 시각
- 마지막 인덱싱 시각
- 상태(active/deprecated/draft)
```

### 홈페이지 크롤링 탭 추가 컬럼

```text
- URL
- 메뉴 경로
- 변경 감지 여부
- 크롤링 성공/실패
- 본문 추출 품질 상태
```

### 법령 탭 추가 컬럼

```text
- 법령명
- 법령 코드
- 시행일
- 개정일
- 현행 버전 여부
```

### 민원편람 탭 추가 컬럼

```text
- 민원명
- 담당부서
- 처리기간
- 수수료
- 온라인 신청 가능 여부
```

# 7. 왜 “RAG 품질은 Admin UI에서 결정된다”는가

이유는 아래와 같습니다.

## 7-1. 오답은 자동으로 안 고쳐짐

모델이 좋아도

* 문서가 없으면 못 맞춤
* 구버전 문서면 틀림
* 내부문서를 써도 안 됨

이걸 잡는 곳이 Admin입니다.

## 7-2. 실제 개선은 운영 루프로 일어남

품질 향상 흐름은 보통 이렇습니다.

```text
질문 발생
→ 미응답 감지
→ QA 분류
→ FAQ 또는 문서 보강
→ 재인덱싱
→ 재측정
```

즉 개선은 모델이 아니라 운영 UI에서 굴러갑니다.

## 7-3. 공공기관은 출처와 최신성이 핵심

그래서

* 문서 최신성 화면
* 법령 개정 반영 화면
* 출처 누락 검수 화면
  이 반드시 있어야 합니다.
