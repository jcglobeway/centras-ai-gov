# 공공기관용 RAG 챗봇 운영 플랫폼 — PRD

> v1.0 · 통합 제품 요구사항 정의서

---

## 1. 제품 정의

공공기관용 RAG 챗봇 운영 플랫폼은 **운영사와 고객사(공공기관)가 역할에 맞게 분리된 어드민**을 통해 챗봇을 운영하는 통합 관제 시스템이다.

**운영사**: 장애를 빨리 찾고, 적은 인력으로 많은 기관을 관리한다.
**고객사**: 챗봇이 민원응대 업무에 얼마나 기여했는지 확인한다.

> 핵심 원칙: 운영사 화면(장애·품질·RAG 운영)과 고객사 화면(민원성과·업무효율·지식관리)은 같은 플랫폼 안에서 권한에 따라 완전히 분리된다.

---

## 2. 설계 이중 축

| 축 | 대상 | 핵심 질문 | 주요 지표 |
|----|------|-----------|-----------|
| **운영사 축** | ops_admin, super_admin | "시스템이 잘 돌고 있는가?" | SLA 준수율, 에러율, 인덱싱 성공률, MTTR |
| **고객사 축** | client_org_admin, client_viewer | "업무 성과가 났는가?" | 자동응대 완료율, 상담 전환율, 만족도, 재문의율 |

두 축에서 동일한 지표가 필요한 경우에도 **보여주는 컨텍스트와 액션이 달라야 한다**.

예: 미응답률
- 운영사: "어느 기관에서 급증했는가?" → 장애/RAG 이슈 탐지
- 고객사: "어떤 민원 카테고리에서 많은가?" → 지식 공백 파악

## 3. 어드민 포털 구조

하나의 플랫폼 안에서 권한에 따라 3개 포털이 분기된다.

### 3-1. Ops Admin (운영사 통합 관제)

역할: 전체 기관 관제, RAG 인덱싱 관리, 장애 대응, 비용/SLA 관리

| 메뉴 | 핵심 화면 |
|------|-----------|
| 운영 대시보드 | 기관 헬스맵 / 이상탐지 / 오늘의 이슈 |
| 기관/고객사 관리 | 기관 목록 / 기관 상세 / SLA 설정 |
| RAG 운영 관리 | 수집·파싱·임베딩·인덱싱 단계별 현황 / 최신화 누락 문서 |
| 품질 모니터링 | 미응답·오답 의심·저평가 응답 현황 |
| 장애/이슈 관리 | 실시간 장애 / 기관 영향도 / 운영 티켓 |
| 배포/버전 관리 | 모델·프롬프트·검색 설정 버전 이력 |
| 비용/사용량 | 기관별 토큰 사용량 / 질문당 평균 비용 |

MVP 우선순위: 운영 대시보드 → RAG 운영 → 품질 모니터링 → 장애/이슈

### 3-2. Client Admin (고객사 성과 대시보드)

역할: 기관별 민원응대 성과 확인, 지식 현황, 챗봇 기본 설정

| 메뉴 | 핵심 화면 |
|------|-----------|
| 기관 대시보드 | 성과 요약 / 자동응대율 / 개선 포인트 |
| 민원응대 성과 | 유형별 해결률 / 1회 해결률 / 업무시간 외 응대 |
| 이용 현황 분석 | 질문량 추이 / 채널별 / 인기 질문 TOP N |
| 실패/전환 분석 | 상담 전환 다발 질문 / 미응답 / 지식 공백 |
| 지식 현황 | 문서 현황 / 최신화 필요 문서 / 문서 추가 요청 |
| 기관 리포트 | 주간·월간 성과 리포트 다운로드 |

MVP 우선순위: 기관 대시보드 → 민원응대 성과 → 실패/전환 분석

### 3-3. QA Admin (품질·지식관리)

역할: 오답 검수, 지식 보강, 승인 워크플로우 (운영사·고객사 담당자 공유)

| 메뉴 | 핵심 화면 |
|------|-----------|
| 검수 대시보드 | 미처리 건수 / 오답 의심 / 고위험 응답 현황 |
| 미응답/오답 관리 | 실패 질문 목록 / 원인 코드 분류 / 처리 상태 |
| 질문 클러스터 | 유사 질문 묶음 / 반복 민원 그룹 / 급증 질문 |
| 문서/FAQ 관리 | 문서 업로드 / FAQ 작성·승인 / 문서 버전 이력 |
| 승인 워크플로우 | 문서 수정 → 검수 → 승인 → 반영 |
| 정책/금칙어 | 민감 카테고리 정책 / 응답 제한 규칙 |

### 3-4. 권한 체계

```text
super_admin       전체 기관/시스템 접근. 모든 포털.
ops_admin         운영 대시보드·RAG·장애·품질 현황. 고객사 설정 일부 제한.
qa_manager        검수·오답·문서 관리·수정 승인 가능.
client_org_admin  자기 기관 대시보드·성과·지식 현황·기관 사용자 관리.
client_viewer     자기 기관 조회 전용. 리포트 다운로드. 설정 변경 불가.
knowledge_editor  문서 업로드·FAQ 작성. 승인 없이 배포 불가.
```

## 4. 지표 6개 레이어

지표는 의사결정 층위에 따라 6개 레이어로 분류한다. 레이어별로 주요 수신자가 다르다.

| # | 레이어 | 주요 수신자 | 대표 지표 |
|---|--------|-------------|-----------|
| 1 | **서비스 이용** | 공통 | 총 질문 수, 순 사용자 수, 일/주/월 추이, 재방문율 |
| 2 | **민원응대 성과** | 고객사 | 자동응대 완료율, 상담 전환율, 1회 해결률, 재문의율, 업무시간 외 응대 비율 |
| 3 | **답변 품질** | 운영사·QA | 답변 성공률, 미응답률, 만족도, 근거 제시율, 환각 의심 비율, 최신 문서 기준 답변 비율 |
| 4 | **RAG 운영** | 운영사 | 인덱싱 성공률, 검색 성공률, 검색 무결과율, 문서 버전 최신화율, 오래된 문서 참조 비율 |
| 5 | **시스템 운영** | 운영사 | API 응답시간, LLM 응답시간, 에러율, SLA 준수율, MTTR, 토큰 사용량 |
| 6 | **운영 업무** | 운영사 | 이슈 처리 소요 시간, 재발 이슈 비율, 검수 대기 건수, 월별 운영 공수 |

## 5. KPI 체계 (2계층)

### 운영사 KPI (기술 운영 관점)

| KPI | 목표 | 설명 |
|-----|------|------|
| SLA 준수율 | > 99.5% | 계약 응답시간 내 처리 비율 |
| E2E 응답시간 P95 | < 3s | 전체 응답 지연 95퍼센타일 |
| 에러율 | < 0.5% | 4xx/5xx/timeout 합산 |
| 인덱싱 성공률 | > 98% | 수집된 문서 중 인덱싱 완료 비율 |
| 미응답률 | < 5% | no_answer + fallback 비율 |
| 이슈 평균 처리 시간(MTTR) | < 4h | 장애 인지 후 복구까지 |
| 기관별 품질 이상 조기탐지율 | > 80% | 임계값 알림으로 선제 탐지 비율 |

### 고객사 KPI (민원응대 성과 관점)

| KPI | 목표 | 설명 |
|-----|------|------|
| 자동응대 완료율 | > 70% | 사람 연결 없이 챗봇으로 종료된 비율 |
| 상담 전환율 | < 20% | 상담원/콜센터로 넘어간 비율 |
| 1회 해결률(FCR) | > 65% | 한 번의 대화에서 해결된 비율 |
| 사용자 만족도 | > 4.0 / 5.0 | 명시적 피드백 평균 |
| 재문의율 | < 10% | 동일/유사 질문이 재유입되는 비율 |
| 업무시간 외 응대 비율 | 측정값 공개 | 야간/주말 자동 처리 비율 |

### RAG 내부 기술 벤치마크 (운영사 내부용)

운영사가 RAG 파이프라인 품질을 자동 측정하는 내부 지표. 고객사에 직접 노출하지 않는다.

| 지표 | 목표 |
|------|------|
| Faithfulness | > 0.90 |
| Recall@K | > 0.85 |
| Precision@K | > 0.70 |
| NDCG@K | > 0.80 |
| Context Relevance | > 0.75 |
| Hallucination Rate | < 5% |
| TTFT (Time to First Token) | < 800ms |

| A06 | 생성 답변 왜곡 (환각·출처 이탈) | 운영사 (프롬프트 수정) |

## 7. 공공기관 특수 요건

일반 기업 챗봇과 달리 공공기관에서 반드시 고려해야 하는 3가지 요건이다.

### 7-1. 최신성 관리

법령·조례·지침·공고는 수시로 변경된다. 폐기 문서를 참조하는 답변은 민원 오류로 직결된다.

- 필수 지표: 문서 최신 업데이트 일자, 폐기 문서 참조 비율, 변경 후 반영 소요 시간
- 필수 뷰: 최신화 위험 알림, 최근 개정 문서와 영향 질문 목록

### 7-2. 설명 가능성

공공기관 답변은 근거 제시가 필수다.

- 필수 지표: 출처 포함 답변 비율, 출처 클릭률, 출처 없는 답변 비율
- 필수 뷰: 답변별 참조 문서 + 문단 매핑

### 7-3. 민감업무 통제

복지·세금·민원 신고·자격요건 오류는 행정 피해로 이어진다.

- 필수 지표: 고위험 민원 카테고리별 상담 전환율, 정책 제한 응답 발생 건수
- 필수 뷰: 고위험 질문 전용 검수 대시보드, 민감 카테고리 별도 승인 워크플로우

## 8. 해결율 측정 방법론

상담사 연결 기능이 없는 환경에서는 해결율을 **단일 숫자**가 아닌 3단계로 구분한다.

| 구분 | 정의 | 산출 방법 |
|------|------|-----------|
| **확정 해결율** | 사용자가 "해결됐어요" 피드백을 남긴 비율 | 해결 응답 수 / 전체 피드백 응답 수 |
| **추정 해결율** | 행동 신호 기반 간접 추정 | 답변 후 재질문 없이 종료 + 관련 링크 클릭 후 이탈 + 동일 질문 미재유입 가중 합산 |
| **미해결 의심률** | 포기·재질문·저만족 신호 | 포기율 + 재문의율 + 저만족 피드백 비율 |

실무 운영: 확정 해결율 + 추정 해결율을 나란히 공개하고, 피드백 응답률을 함께 제시한다.

## 9. 핵심 화면 설계 원칙

### 원칙 1: 모든 지표는 "조치 가능"해야 한다

지표 노출만으로는 충분하지 않다. 지표 옆에 즉시 실행 가능한 액션을 배치한다.

```
미응답률 8%
  → [미응답 질문 보기]  [문서 추가 요청]  [FAQ 생성]
```

### 원칙 2: 운영사 / 고객사 화면은 분리한다

같은 데이터도 수신자에 따라 다르게 표현한다.

```
운영사: 에러율 / 인덱싱 상태 / 모델 응답시간
고객사: 민원 해결률 / 상담 전환율 / 만족도
```

### 원칙 3: 지표는 "비교 가능"해야 한다

절대값이 아닌 전월·전주 대비 변화율을 기본 표시한다.

```
자동응대율 68%  (전월 62%  +6% 개선)
```

### 원칙 4: 모든 상세 화면은 공통 필터를 지원한다

최소 필터: 기관 / 기간 / 채널 / 민원 카테고리 / 응답 결과 / 문서 유형

## 11. 데이터 키 설계 원칙

모든 이벤트 로그에 공통 키를 표준화하지 않으면 기관별 비교와 성과 측정이 불가능하다.

| 키 | 설명 |
|----|------|
| `org_id` | 기관 식별자 (모든 데이터의 스코프 기준) |
| `service_id` | 채널/서비스 식별자 |
| `session_id` | 대화 세션 |
| `question_id` | 개별 질문 |
| `user_anon_id` | 익명 사용자 식별자 |
| `asked_at` | 질문 시각 |
| `question_category` | 민원 카테고리 분류 결과 |
| `answer_result_code` | `answered` / `no_answer` / `fallback` / `error` |
| `failure_code` | A01~A10 실패 원인 코드 |
| `escalated_to_human` | 상담 전환 여부 |
| `satisfaction` | 사용자 피드백 점수 |
| `source_doc_ids` | 참조 문서 ID 목록 |
| `doc_version` | 참조 문서 버전 |
| `model_version` | 사용된 LLM 버전 |
| `search_score` | 검색 유사도 점수 |
| `response_time_ms` | 전체 응답 지연 (ms) |

## 12. 품질 자동화 방법론

운영사가 RAG 파이프라인 품질을 자동으로 측정하는 방법론. Section 5의 RAG 내부 기술 벤치마크를 산출하는 도구다.

### LLM-as-Judge

Claude/GPT를 평가자로 활용해 Faithfulness·Relevance·Completeness를 자동 채점. Human 레이블링 없이 대규모 평가 가능.

```
평가 항목 (각 0-1점):
  1. Faithfulness:   답변이 컨텍스트에만 근거하는가?
  2. Relevance:      답변이 질문에 적절한가?
  3. Completeness:   답변이 충분히 완전한가?
```

### RAGAS 자동 평가

faithfulness, answer_relevancy, context_recall, context_precision을 한 번에 산출한다. Synthetic QA 데이터셋과 결합하면 Human 레이블링 없이 지속적 회귀 테스트가 가능하다.

### Synthetic QA Ground Truth

LLM으로 문서 기반 Q&A 쌍을 자동 생성한다. 난이도별(쉬움/중간/어려움) + 질문 유형별(사실 조회/절차 안내/자격 확인/비교/복합) 균등 샘플링으로 편향 없는 벤치마크를 구성한다.

### 온라인 피드백 루프

| 신호 유형 | 신호 | 가중치 |
|-----------|------|--------|
| 명시적 | 좋아요 | +1.0 |
| 명시적 | 싫어요 | -1.0 |
| 암묵적 | 응답 복사 | +0.5 |
| 암묵적 | 재질문 발생 | -0.3 |
| 암묵적 | 답변 후 세션 종료 | +0.5 |

## 13. 구현 로드맵

### Phase 1 — 최소 운영 체계 (기본 지표 + 헬스 대시보드)

**목표**: 서비스가 돌아가고, 기본 성과가 보이게

수집 필수: 질문/답변 로그, 세션 로그, 사용자 피드백, 문서 인덱싱 로그, 에러 로그

제공 화면:
- 운영사: 헬스 대시보드, 인덱싱 현황, 장애 현황
- 고객사: 문의량, 자동응대율, 만족도, 상위 질문

### Phase 2 — 품질 관리 체계 (실패 코드 + 검수 워크플로우)

**목표**: 답변 품질을 개선할 수 있게

추가 수집: 답변 근거 문서, 검색 후보군, 신뢰도 점수, **실패 원인 코드(A01-A10)**, 질문 카테고리/의도 분류 결과

추가 화면: 실패 응답 원인 분석, 지식 공백 분석, 고위험 질문 모니터링, 품질 검수 큐

자동화: RAGAS + LLM-as-Judge 자동 평가 파이프라인 구동

### Phase 3 — 고객사 성과관리 체계

**목표**: 챗봇이 행정업무에 얼마나 기여했는지 증명

추가 수집: 민원 유형 매핑, 재문의 여부, 업무시간 외 응대 데이터, 부서/업무 단위 성과 데이터

추가 화면: 민원 카테고리별 성과, 상담 절감 효과, 기간별 개선 추이, 부서별 활용도

### Phase 4 — 고도화·자동화

**목표**: 운영 공수를 줄이고 선제 대응

추가 기능: 이상탐지 알림, 자동 재인덱싱, 문서 최신화 알림, 실패 질문 자동 클러스터링, 기관별 품질 리포트 자동 생성
