# Ops 포털 메뉴 설계 — 목적과 핵심

세 메뉴 각각의 목적과 핵심을 짚어드릴게요.

---

## 프롬프트 엔지니어링 — 핵심

**목적:** 챗봇의 성격과 답변 방식을 정의하는 것. 가장 직접적으로 답변 품질에 영향.

가장 중요한 것은 **버전 이력 + 롤백**입니다.

프롬프트는 조금만 바꿔도 Faithfulness, Hallucination Rate가 크게 출렁입니다. 그래서 "잘 되던 버전으로 즉시 되돌릴 수 있는가"가 편집 기능 자체보다 더 중요해요. 프롬프트를 잘못 수정했을 때 롤백이 없으면 이전 상태를 기억해서 수동으로 복원해야 하는데, 실수가 날 수밖에 없습니다.

시스템 프롬프트 편집기는 기본이고, **변경 → 시뮬레이션 룸 검증 → 배포**로 이어지는 워크플로가 지켜지는지가 그 다음으로 중요합니다. 즉시 반영이 안 되게 막는 것 자체가 사고 예방입니다.

---

## RAG 파라미터 튜닝 — 핵심

**목적:** 검색 품질과 비용의 균형을 조정하는 것.

가장 중요한 것은 **Cosine Similarity threshold(유사도 임계값)**입니다.

이 값 하나가 Recall@K, Precision@K, Hallucination Rate, 비용을 동시에 좌우합니다. 너무 낮게 설정하면 관련 없는 청크를 가져와서 Hallucination이 올라가고, 너무 높게 설정하면 정답 청크를 놓쳐서 Recall이 떨어집니다. Top-K와 Reranker 토글은 임계값 설정 이후에 미세 조정하는 수준이에요.

그 다음은 **변경 전후 Recall@K 시뮬레이션 미리보기**입니다. 슬라이더를 움직일 때 결과가 어떻게 바뀌는지 바로 보여주지 않으면, 운영자가 감으로 조정할 수밖에 없어요.

---

## 모델 서빙 관리 — 핵심

**목적:** LLM API 연결 상태와 리소스를 모니터링하는 것. 설정보다 **상태 확인** 성격이 강함.

가장 중요한 것은 **LLM API 연동 상태**입니다.

API가 끊기거나 응답이 느려지면 시스템 전체가 멈춥니다. 다른 지표(CPU, Memory)는 여유가 있어도 API 연결 하나가 끊기면 Answer Rate가 즉각 0%로 떨어지는 가장 단일 장애점(SPOF)입니다.

Rate Limit 히트율이 그 다음입니다. Rate Limit에 자주 걸리면 응답 지연이 생기고 사용자 경험이 나빠지는데, 원인을 모르면 Latency P95가 왜 올라가는지 찾기 어렵습니다. 그래서 Ops 팀이 Latency 이상을 감지했을 때 Rate Limit 히트율을 보조 지표로 같이 확인합니다.

인스턴스 리소스(CPU/Memory/GPU)는 중요하지만 이미 인프라 모니터링 툴(Grafana 등)에서 보는 경우가 많아서 어드민에서는 참고 수준입니다.

## 요약

| 메뉴 | 제일 중요한 것 | 이유 |
|------|--------------|------|
| 프롬프트 엔지니어링 | **버전 이력 + 롤백** | 잘못된 변경을 즉시 되돌릴 수 있어야 함. 편집보다 복구가 더 중요 |
| RAG 파라미터 튜닝 | **Cosine Similarity threshold** | Recall, Precision, Hallucination, 비용을 동시에 결정하는 핵심 변수 |
| 모델 서빙 관리 | **LLM API 연동 상태** | 단일 장애점. API 끊기면 시스템 전체 중단 |
