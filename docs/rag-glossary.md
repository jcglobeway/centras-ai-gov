## ① RAG 파이프라인 기본 개념

### 🔗 RAG — 검색 증강 생성

> **AI가 답하기 전에 관련 문서를 먼저 찾아보는 방식**

**쉬운 설명**  
Retrieval-Augmented Generation의 약자. ChatGPT 같은 AI는 학습 당시의 지식만 알지만, RAG는 질문을 받으면 먼저 우리 회사 문서·DB를 검색해서 관련 내용을 찾아온 뒤 그걸 참고해 답변합니다. 덕분에 최신 정보나 사내 정보도 답할 수 있습니다.

**🔍 비유로 이해하기**  
오픈북 시험. AI가 시험을 볼 때 교재를 펴볼 수 있게 한 것. 교재(문서)를 잘 찾을수록 정답률이 높아집니다.

| | |
|---|---|
| ✅ 좋은 상태 | 질문과 관련된 문서를 잘 찾고, 그 내용에 근거해 정확하게 답변 |
| ❌ 나쁜 상태 | 관련 없는 문서를 가져오거나, 문서 내용과 다른 말을 지어냄 |

---

### ✂️ Chunk (청크) — 문서 조각

> **긴 문서를 검색하기 좋게 잘라낸 작은 단위**

**쉬운 설명**  
문서 전체를 통째로 AI에 넣으면 너무 길어서 처리하기 어렵습니다. 그래서 문서를 500~1,000자 단위로 잘라 청크로 만들고, 질문이 들어오면 관련 청크만 골라서 사용합니다.

**🔍 비유로 이해하기**  
백과사전을 항목별로 분리한 카드 인덱스. 필요한 카드만 꺼내 쓰는 방식.

| | |
|---|---|
| ✅ 좋은 상태 | 청크 크기가 적절해서 핵심 내용이 잘 담겨 있음 |
| ❌ 나쁜 상태 | 너무 짧으면 맥락이 잘리고, 너무 길면 검색 정확도가 떨어짐 |
| 🎯 목표 | 500~1,000 토큰 권장 |

---

### 📍 Embedding (임베딩) — 의미 좌표화

> **텍스트의 "의미"를 숫자 벡터로 변환하는 과정**

**쉬운 설명**  
"사과"와 "애플"은 표현은 다르지만 의미가 비슷합니다. 임베딩은 이런 의미적 유사함을 수백 개의 숫자(벡터)로 표현합니다. 벡터 공간에서 비슷한 의미일수록 가까이 위치하기 때문에, "거리 계산"만으로 의미 검색이 가능해집니다.

**🔍 비유로 이해하기**  
지도 위 좌표. "서울"과 "인천"은 좌표가 가깝고, "뉴욕"은 멀리 있는 것처럼, 의미가 비슷한 문장은 벡터 공간에서 가깝게 위치합니다.

| | |
|---|---|
| ✅ 좋은 상태 | 질문과 관련 문서의 벡터가 가까워 빠르고 정확한 검색 가능 |
| ❌ 나쁜 상태 | 임베딩 모델이 맞지 않으면 유사한 내용도 멀게 인식됨 |

---

### 🗄️ Vector DB — 의미 검색 데이터베이스

> **임베딩 벡터를 저장하고 빠르게 유사도 검색하는 전용 DB**

**쉬운 설명**  
일반 DB는 "정확히 일치하는 값"을 찾습니다. 벡터 DB는 "비슷한 의미"를 찾습니다. 수십만 개의 청크 벡터를 저장해 두고, 새 질문이 들어오면 가장 의미적으로 가까운 청크들을 밀리초 안에 반환합니다. Qdrant, Weaviate, Pinecone 등이 대표적입니다.

**🔍 비유로 이해하기**  
음악 스트리밍 앱의 "비슷한 노래 추천". 노래를 벡터로 표현해 취향이 가까운 곡을 찾아줍니다.

| | |
|---|---|
| ✅ 좋은 상태 | 빠른 응답(< 100ms)으로 관련 청크 정확히 반환 |
| ❌ 나쁜 상태 | 인덱스가 오래됐거나 메모리 부족 시 검색 품질 저하 |
| 🎯 목표 | QPS 안정, Memory < 80% |

## ② Retrieval (검색) 지표

### 🎯 Recall@K — 재현율

> **정답 문서가 검색 결과 상위 K개 안에 들어왔는가?**

**쉬운 설명**  
K=5라면 "검색 결과 1~5위 안에 정답 문서가 포함돼 있는지"를 측정합니다. 100개 질문 중 88개에서 정답 문서가 Top-5 안에 있었다면 Recall@5 = 0.88입니다. 정답을 놓치지 않는 능력입니다.

**🔍 비유로 이해하기**  
시험에서 정답지를 가져왔는가? K개의 후보 중 진짜 정답이 포함됐는지 확인.

| | |
|---|---|
| ✅ 좋은 상태 | 정답 문서를 빠짐없이 가져옴 → 좋은 답변 가능 |
| ❌ 나쁜 상태 | 정답 문서를 가져오지 못함 → 아무리 좋은 AI도 맞는 답을 못 함 |
| 🎯 목표 | > 0.85 |

---

### 🔬 Precision@K — 정밀도

> **가져온 K개 문서 중 실제로 관련 있는 게 몇 개인가?**

**쉬운 설명**  
K=5로 5개 문서를 가져왔는데 그 중 3개만 실제 관련 있다면 Precision@5 = 0.60입니다. Recall이 "빠짐없이 가져왔나"라면, Precision은 "쓸데없는 걸 안 가져왔나"를 봅니다.

**🔍 비유로 이해하기**  
낚시할 때 그물에 걸린 것 중 먹을 수 있는 물고기 비율. 쓰레기가 많이 걸릴수록 Precision이 낮아집니다.

| | |
|---|---|
| ✅ 좋은 상태 | 가져온 문서 대부분이 질문에 관련 있음 → 답변 품질 향상 |
| ❌ 나쁜 상태 | 관련 없는 문서가 많이 섞임 → AI가 헷갈려서 엉뚱한 답 가능 |
| 🎯 목표 | > 0.70 |

---

### 🥇 MRR (Mean Reciprocal Rank) — 평균 역순위

> **정답 문서가 검색 결과에서 평균적으로 몇 번째에 위치하는가?**

**쉬운 설명**  
정답이 1위면 점수 1.0, 2위면 0.5, 3위면 0.33… 이 점수의 평균이 MRR입니다. Recall은 "Top-K 안에만 있으면 됨"이지만, MRR은 "더 앞 순위에 있을수록 좋음"을 반영합니다.

**🔍 비유로 이해하기**  
검색엔진에서 원하는 결과가 첫 페이지 맨 위에 뜰수록 좋은 것처럼, 정답이 1등이어야 가장 점수가 높습니다.

| | |
|---|---|
| ✅ 좋은 상태 | 정답 문서가 항상 1~2위 → MRR 0.9 이상 |
| ❌ 나쁜 상태 | 정답이 자주 하위 순위에 밀림 → 앞 순위 노이즈가 답변 품질 저하 |
| 🎯 목표 | > 0.75 |

---

### 📐 Cosine Similarity — 코사인 유사도

> **질문과 문서 벡터가 얼마나 같은 방향을 가리키는가 (0~1)**

**쉬운 설명**  
두 벡터의 각도를 측정합니다. 완전히 같은 방향이면 1.0(완전 일치), 직각이면 0(무관계)입니다. 벡터 검색에서 이 값이 클수록 의미적으로 유사한 문서입니다. 보통 0.7 이상이면 관련 있다고 봅니다.

**🔍 비유로 이해하기**  
두 사람이 같은 방향을 바라보면 유사도가 높은 것. 정반대를 보면 유사도가 낮습니다.

| | |
|---|---|
| ✅ 좋은 상태 | 검색된 청크의 유사도가 0.7 이상 → 관련도 높음 |
| ❌ 나쁜 상태 | 유사도가 0.5 미만인 청크가 많음 → 엉뚱한 내용 검색 |
| 🎯 목표 | Top-K 평균 > 0.65 |

## ③ Generation (생성) 지표

### 📖 Faithfulness — 충실도

> **답변이 검색된 문서에 근거하고 있는가? (지어내지 않았는가?)**

**쉬운 설명**  
AI가 검색해온 문서 내용에만 근거해서 답했는지를 측정합니다. 문서에 없는 내용을 스스로 지어냈다면 Faithfulness가 낮습니다. LLM Judge가 "이 답변의 각 주장이 문서에 있는가?"를 자동으로 채점합니다.

**🔍 비유로 이해하기**  
독후감을 쓸 때 책 내용을 기반으로 쓰면 Faithfulness가 높고, 읽지도 않고 지어내면 낮은 것.

| | |
|---|---|
| ✅ 좋은 상태 | 0.95 → 답변의 95%가 문서에 근거한 내용 |
| ❌ 나쁜 상태 | 0.70 → 30%가 문서 밖 내용 (환각 위험) |
| 🎯 목표 | > 0.90 |

---

### 👻 Hallucination Rate — 환각 비율

> **AI가 사실처럼 그럴듯하게 꾸며낸 내용의 비율**

**쉬운 설명**  
AI는 모르는 게 있을 때 "모른다"고 하지 않고 그럴듯한 내용을 지어내는 경향이 있습니다. 이를 환각(Hallucination)이라 합니다. RAG에서는 검색 문서에 없는 내용을 답변에 포함했을 때 환각으로 판정합니다. Faithfulness의 반대 개념입니다.

**🔍 비유로 이해하기**  
자신 없으면서도 자신 있는 척 틀린 답을 말하는 학생. 모른다고 하면 되는데 지어내는 경우.

| | |
|---|---|
| ✅ 좋은 상태 | < 3% → 거의 지어내지 않음 |
| ❌ 나쁜 상태 | > 5% → 답변 신뢰도 위험 수준 |
| 🎯 목표 | < 5% |

---

### 💬 Answer Relevance — 답변 관련성

> **답변이 질문에 제대로 답하고 있는가?**

**쉬운 설명**  
Faithfulness가 "문서에 근거했는가"라면, Answer Relevance는 "질문에 대한 답을 했는가"를 봅니다. 문서에 있는 내용을 충실히 썼더라도 질문과 관계없는 내용을 답했다면 Relevance가 낮습니다.

**🔍 비유로 이해하기**  
"서울에서 부산까지 KTX 시간이 얼마나 걸려요?"라는 질문에 KTX 역사를 답하면 Faithfulness는 높지만 Relevance는 낮습니다.

| | |
|---|---|
| ✅ 좋은 상태 | > 0.85 → 질문 의도에 맞는 답변 |
| ❌ 나쁜 상태 | < 0.70 → 동문서답, 질문 이해 실패 |
| 🎯 목표 | > 0.85 |

---

### ✅ Answer Correctness — 답변 정확도

> **답변이 실제 정답과 얼마나 일치하는가?**

**쉬운 설명**  
Synthetic QA 데이터셋(사전에 만들어둔 정답 Q&A 쌍)과 RAG의 실제 답변을 비교해 정확도를 측정합니다. Faithfulness·Relevance는 형식적 품질이지만, Correctness는 "실제로 맞는 말을 했는가"를 직접 평가합니다.

**🔍 비유로 이해하기**  
채점 기준표와 학생 답안을 직접 비교하는 것.

| | |
|---|---|
| ✅ 좋은 상태 | > 0.80 → 정답과 거의 일치 |
| ❌ 나쁜 상태 | < 0.65 → 사실과 다른 답변 다수 |
| 🎯 목표 | > 0.80 |

## ④ 시스템 운영 지표

### ⏱️ E2E Latency P95 — 전체 응답시간 95백분위

> **사용자 100명이 쓸 때 가장 느린 5명의 응답시간 기준값**

**쉬운 설명**  
E2E(End-to-End)는 사용자가 질문을 보낸 순간부터 답변을 받는 순간까지의 전체 시간입니다. P95는 상위 5%의 느린 케이스를 기준으로 합니다. 평균보다 P95가 중요한 이유는, 느린 경험을 하는 사람이 실제로 존재하기 때문입니다.

**🔍 비유로 이해하기**  
버스 배차 시간. "평균 10분"이라도 어떤 날은 25분을 기다릴 수 있습니다. P95는 그 "최악에 가까운 경험"을 측정합니다.

| | |
|---|---|
| ✅ 좋은 상태 | 1,500~2,000ms → 쾌적한 대화형 경험 |
| ❌ 나쁜 상태 | > 3,000ms → 사용자가 기다리다 포기 가능 |
| 🎯 목표 | P95 < 3,000ms |

---

### ⚡ Cache Hit Rate — 캐시 적중률

> **이미 처리한 비슷한 질문의 답변을 재활용한 비율**

**쉬운 설명**  
같거나 매우 비슷한 질문이 반복되면 다시 검색·생성하지 않고 저장해둔 답변을 바로 반환합니다. 이를 시맨틱 캐시라 하며, 적중할수록 응답이 빠르고 API 비용이 절감됩니다.

**🔍 비유로 이해하기**  
자주 받는 고객 문의에 미리 준비된 답변 템플릿 활용. 매번 처음부터 작성하지 않아도 됩니다.

| | |
|---|---|
| ✅ 좋은 상태 | > 30% → 비용과 속도 모두 개선 |
| ❌ 나쁜 상태 | < 10% → 캐시 전략 재검토 필요 |
| 🎯 목표 | > 20% |

---

### 🕳️ Knowledge Gap Rate — 지식 공백률

> **사용자가 질문했지만 시스템이 답하지 못한 비율**

**쉬운 설명**  
검색 결과가 없거나 관련도가 너무 낮아서 답변을 생성하지 못한 경우입니다. 이 비율이 높다는 건 지식베이스에 없는 주제로 질문이 들어오고 있다는 신호입니다. 미커버 질문들을 분석하면 다음 콘텐츠 추가 우선순위를 알 수 있습니다.

**🔍 비유로 이해하기**  
"해당 항목 없음"이 뜨는 검색결과 비율. 자주 뜬다면 그 주제의 문서를 추가해야 합니다.

| | |
|---|---|
| ✅ 좋은 상태 | < 5% → 지식베이스가 질문 범위를 충분히 커버 |
| ❌ 나쁜 상태 | > 15% → 지식베이스 확충 시급 |
| 🎯 목표 | < 8% |

### 🔒 PII Leakage — 개인정보 유출

> **AI 답변에 개인정보(주민번호, 연락처 등)가 포함된 경우**

**쉬운 설명**  
지식베이스에 개인정보가 포함된 문서가 있을 때, 검색 결과가 그대로 답변에 노출될 수 있습니다. Presidio 같은 도구로 모든 답변을 실시간 검사해 개인정보를 마스킹하고 이벤트를 기록합니다. 1건이라도 발생하면 즉시 대응이 필요합니다.

**🔍 비유로 이해하기**  
콜센터 직원이 고객의 주민번호를 통화 중에 실수로 말해버리는 상황.

| | |
|---|---|
| ✅ 좋은 상태 | 0건 → 개인정보 유출 없음 |
| ❌ 나쁜 상태 | 1건 이상 → 즉각 대응 필요 (법적 리스크) |
| 🎯 목표 | 0건 |

### 🗓️ Index Freshness — 인덱스 신선도

> **지식베이스가 얼마나 최근 문서로 업데이트돼 있는가**

**쉬운 설명**  
마지막으로 문서를 인덱싱한 이후 경과한 시간입니다. 오래될수록 최신 정보가 반영되지 않아 답변이 오래된 내용을 포함할 수 있습니다. 매일 자동 인덱싱이 되도록 파이프라인을 설정합니다.

**🔍 비유로 이해하기**  
식당 메뉴판. 오래된 메뉴판에는 단종된 메뉴가 그대로 있어 주문 시 혼란이 생깁니다.

| | |
|---|---|
| ✅ 좋은 상태 | < 12h → 거의 실시간 반영 |
| ❌ 나쁜 상태 | > 72h → 오래된 정보로 오답 위험 |
| 🎯 목표 | < 24h |
